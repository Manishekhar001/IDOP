IDOP — Intelligent Data Operations Platform

A complete enterprise platform that lets you ask questions in plain English and get answers from your database, documents, or both — with built-in safeguards for data mutations.

What is IDOP?

IDOP is a FastAPI-based backend that combines three major capabilities into one unified system:

  • Text-to-SQL — Ask "How many customers are there?" and get the answer directly from your database
  • Spreadsheet Mutations — Upload an Excel file to insert, update, or delete rows with business rule validation and human approval
  • Advanced RAG — Upload PDFs and ask questions about them, with automatic web search fallback when documents don't have the answer

All three are connected through a smart 5-path router that automatically decides which pipeline to use based on your question.

Why was IDOP built?

Most business data lives in two places: structured databases (PostgreSQL tables with rows and columns) and unstructured documents (PDFs, TXTs, CSVs with policies and guidelines). Typically you need SQL knowledge for the first and manual searching for the second.

IDOP bridges both worlds by using LLMs to understand natural language, generate the right queries, search the right documents, and synthesize answers — all with human-in-the-loop safeguards for data-modifying operations.

The 5-Path Semantic Router

The router is the brain of IDOP. It uses an LLM to classify every incoming question into one of five paths. Here's how it decides:

SQL

Data Query

Questions about data in database tables (customers, products, orders). E.g. "How many customers are in Canada?"

MUTATION

Data Modification

Requests to insert, update, or delete data. E.g. "Update prices for all products in category X"

RAG

Document Search

Questions about uploaded documents (PDFs, TXTs). E.g. "What does the refund policy say?"

CHAT

General Chat

Greetings, help questions, or anything not needing DB or documents. E.g. "Hello! Who are you?"

HYBRID

Combined SQL + RAG

Questions needing BOTH database data AND document context. E.g. "Are our sales numbers following the discount policy?"

Router Prompt (Exact)

The router uses a specific prompt to classify queries. It analyzes the question and returns one of: SQL, MUTATION, RAG, CHAT, or HYBRID.

Route decisions:
1. SQL: Fetching, querying, summarizing, or analyzing transactional data
   from database tables (products, customers, orders).
2. MUTATION: Performing modifications like inserting, updating, loading or
   deleting rows from tables, especially triggered by spreadsheets.
3. RAG: Searching for knowledge, policies, guidelines, or non-tabular
   instructions in PDFs or TXT documents.
4. CHAT: General conversational phrases, greetings, help questions, system
   status, or other miscellaneous queries.
5. HYBRID: Queries that require BOTH querying transactional database tables
   and searching for knowledge/guidelines in documents.

4. SQLValidator: ✅ SELECT-only, safe
5. LLM Judge: ✅ Correct semantic match
6. Auto-execute: Query runs → returns 42
Answer: "There are 42 customers located in Canada."

Feature 1: Text-to-SQL — Deep Dive

Every component in the SQL pipeline, explained in detail.

1. VannaService — The SQL Generator

Purpose: Converts natural language questions into PostgreSQL SQL queries.

How it works: At startup, VannaService.complete_training() connects to the live database and queries information_schema to build a complete text description of the database schema (all tables, columns, types, foreign keys, constraints). This schema context is stored and used for every query.

Dual-path fallback:

Question

Vanna Agent

Fail?
→
Fallback LLM
Groq llama-3.3
  • Primary (Vanna): Uses OpenAI's API with gpt-4o-mini. Vanna 2.0.2's built-in OpenAILlmService only supports OpenAI. However, Vanna's LlmService is an abstract base class — in theory, a custom adapter (e.g., GroqLlmService) could be written to use any provider.
  • Fallback (Groq): If Vanna fails for any reason, sends the same schema context to get_chat_llm() (Groq's llama-3.3-70b-versatile) with a prompt asking it to generate SQL.

Why Vanna? Vanna provides multi-turn SQL refinement (can have a back-and-forth conversation about SQL), automatic self-correction on errors, and tool execution capabilities. In practice, the Groq fallback generates equally good SQL for single-shot queries.

2. SQLValidator — Safety Guard

Purpose: Prevents execution of dangerous SQL commands.

Forbidden commands (blocked):

CommandWhy Blocked
DROP, TRUNCATE, ALTERSchema modification — could delete entire tables
GRANT, REVOKEPermission changes — security risk
CREATE, REPLACEObject creation — could overwrite existing objects
DELETE, UPDATE, INSERTData modification — Feature 2 (Mutations) handles these
EXECUTE, EXECDynamic execution — SQL injection risk
COMMIT, ROLLBACKTransaction control — could break database state

Enforcement: Only SELECT queries are permitted in Feature 1. The validator uses word-boundary regex to prevent false positives (e.g., "updated_at" column name won't trigger the UPDATE block).

3. LLMJudge — Semantic Auditor

Purpose: An LLM-as-Judge that checks if the generated SQL semantically matches the user's question.

Evaluation criteria:

  1. Are the correct tables joined using correct keys?
  2. Are filters applied on correct columns/segments?
  3. Does it prevent hallucinations (correct column field names)?

Uses get_memory_llm() (lighter model) for cost efficiency. If the judge fails (API error), it defaults to is_correct=True to avoid blocking valid queries.

4. ApprovalGate — Cryptographic Tokens

Purpose: Provides secure single-use tokens for human-in-the-loop approval.

Uses Python's secrets.token_hex(32) (64-character hex string). Tokens are:

  • Persisted in PostgreSQL table idop_approval_tokens
  • Also stored in-memory as fallback (for tests / database outages)
  • Verified using secrets.compare_digest() (timing-safe comparison)
  • Deleted after use — strictly single-use

5. SQLExecutor — Query Runner

Purpose: Executes approved SQL against PostgreSQL and logs to audit table.

Every execution is logged to idop_audit_logs with: query_id, original question, SQL text, status (SUCCESS/FAILED), and timestamp. Both successful and failed executions are recorded.

Feature 2: Spreadsheet Mutations — Deep Dive

Every component in the mutation pipeline, explained in detail.

1. FileParser — Spreadsheet Reader

Purpose: Reads Excel (.xlsx/.xls) and CSV files into Python dictionaries.

Uses pandas.read_csv() and pandas.read_excel(). Converts NaN/None values to standard Python None. Warns if file exceeds 50MB. Max row count: 1000 (configurable via API parameter or business_rules/rules.json).

2. ColumnMapper — Semantic Column Matching

Purpose: Uses LLM to map messy spreadsheet column headers to formal database column names.

Example: A spreadsheet column "Unit Price (USD)" is semantically mapped to the database column "price". The LLM receives both the file headers and the target DB columns (from TABLE_SCHEMAS in schema_registry.py) and returns a JSON mapping.

If the LLM fails, it falls back to basic string normalization (lowercase, underscores).

3. OpClassifier — Operation Type Classifier

Purpose: Classifies the user's intent as INSERT, UPDATE, or DELETE.

Uses an LLM with a simple prompt: "Analyze the user's intent to determine if they want to insert new rows, update existing rows, or delete rows." Default fallback: INSERT.

4. RuleValidator — Business Rules Engine

Purpose: Validates each row against configurable business rules.

Rules are defined in business_rules/rules.json and support:

Rule TypeDescriptionExample
numericNumeric range check (min/max)price > 0
integerInteger range check (min/max)stock_quantity >= 0
regexPattern matchemail format validation
enumAllowed values checksegment in [SMB, Enterprise, Individual]

5. MutationGenerator — SQL Builder

Purpose: Generates parameterized SQL statements for bulk operations.

  • INSERT: Single bulk statement with all rows as parameters — INSERT INTO table (col1, col2) VALUES (%s, %s)
  • UPDATE: Individual statements per row — UPDATE table SET col1 = %s WHERE id = %s
  • DELETE: Single bulk statement — DELETE FROM table WHERE id IN (%s, %s, ...)

All SQL is parameterized to prevent SQL injection.

6. MutationExecutor — Atomic Transaction Runner

Purpose: Executes mutations inside an all-or-nothing database transaction.

Critical safety guarantee: If ANY row fails, ALL changes are rolled back. The database remains in its original state. Uses conn.autocommit = False and explicit conn.commit() / conn.rollback().

Feature 3: Advanced RAG — Deep Dive

Every component in the RAG pipeline, explained in detail.

1. RetrievalService — Document Finder

Purpose: Searches Qdrant vector database for relevant document chunks.

Three search modes:

ModeMethodBest For
denseSemantic vector similarity (embeddings)Conceptual matches, synonyms
sparseKeyword BM25 via SparseVectorExact term matches, proper nouns
hybridRRF fusion of dense + sparse resultsBest overall recall

2. HyDE — Hypothetical Document Embeddings

Purpose: Improves retrieval by first generating a hypothetical document that would answer the question, then searching with that instead of the original query.

Example: For "What is the refund policy?", HyDE generates: "The refund policy states that customers can return products within 30 days for a full refund. For delivered orders, refunds are processed back to the original payment method within 5 business days." This hypothetical passage is more similar to real documents than the short question, leading to better search results.

3. SparseVectorService — BM25 Keyword Search

Purpose: Generates sparse vectors for keyword-based search, complementing dense semantic search.

Tokenizes text (lowercase, removes stop words), counts term frequencies, hashes each token to a 32-bit integer index. Creates Qdrant SparseVector with (indices, values) format. The stop word list includes 59 common English words.

4. RerankingService — Precision Booster

Purpose: Uses Voyage AI's cross-encoder reranker to re-score retrieved chunks for higher precision.

Unlike bi-encoders (which encode query and document separately), cross-encoders process query+document pairs together, producing more accurate relevance scores. Uses rerank-2.5 model from Voyage AI.

5. ContextEnrichment — Window Expansion

Purpose: Fetches neighboring chunks (before and after) each retrieved chunk to provide surrounding context.

For each retrieved chunk at index X, it fetches chunks at X-1 and X+1 (configurable neighbor count), concatenates them with overlap handling, creating a wider context window. This helps the LLM understand the full context rather than isolated snippets.

6. CRAGEvaluator — Corrective RAG

Purpose: Prevents the LLM from using irrelevant documents by scoring each chunk's relevance.

Each chunk is scored by an LLM on a 0.0–1.0 scale. Based on thresholds (upper: 0.7, lower: 0.3), the system decides whether to use docs, supplement with web search, or use web search exclusively.

7. WebSearchService — Tavily Web Search

Purpose: Fetches real-time information from the web when documents don't have the answer.

Uses Tavily AI API (purpose-built for LLM web search). The query is first rewritten by an LLM into a focused 6–14 keyword search query. Results include title, URL, and content snippet.

8. ContextRefinement — Sentence Filtering

Purpose: Removes irrelevant sentences from the retrieved context before sending to the LLM.

Decomposes all retrieved text into sentences, sends them to an LLM with the question, asks which sentences directly help answer the question. Only kept sentences are included in the final context.

9. SRAGVerifier — Self-Reflective RAG

Purpose: Prevents hallucination by verifying the answer against the context.

Three-stage verification:

  1. Support check: Is every claim in the answer backed by the context? (fully_supported / partially_supported / no_support). If not fully supported, revises the answer (up to 2 retries).
  2. Revise: Rewrites the answer to remove unsupported claims, keeping only what's backed by context.
  3. Usefulness check: Is the answer actually useful to the user? If not useful, rewrites the search query and retries (up to 2 retries).

10. RAGASEvaluator — Quality Metrics

Purpose: Computes RAGAS-style evaluation metrics without external libraries.

Three metrics computed by LLM:

MetricWhat It Measures
Answer RelevancyDoes the answer directly address the question? (0.0–1.0)
FaithfulnessAre all claims in the answer supported by context? (0.0–1.0)
Context PrecisionHow many retrieved chunks are actually relevant? (0.0–1.0)

Memory System

IDOP uses a dual memory system for personalized conversations.

STM

Short-Term Memory

What it stores: Conversation history within a thread

Storage: Postgres via AsyncPostgresSaver (LangGraph checkpointer)

Summarization: When a thread exceeds 6 messages (configurable), the LLM summarizes older messages and deletes them, keeping only the summary + last 2 messages

LTM

Long-Term Memory

What it stores: User profile facts that persist across sessions

Storage: Postgres via AsyncPostgresStore (LangGraph store)

Extraction: After every message, LTMService uses an LLM to extract new facts (identity, preferences, projects, tools). Only genuinely NEW facts are stored (deduplication against existing memories)

Example fact: "User is a Sales Manager specializing in SMB segments"

Memory Namespace Structure (LangGraph AsyncPostgresStore)

LangGraph's AsyncPostgresStore organizes data by namespace tuples (prefix, user_id, kind). IDOP uses:

LTM namespace: ("user", "{user_id}", "details")
Each item key: UUID v4
Each item value: {"data": "User prefers CSV files reports"}

Retry Logic

Both embed_query and embed_documents are wrapped with exponential backoff retry for HTTP 429 (rate limit / quota) errors:

  • Max retries: 3
  • Base delay: 2 seconds
  • Backoff multiplier: 4x (2s → 8s → 32s)
  • After all retries: raises EmbeddingQuotaError

Why Groq?

Groq offers free-tier access to Llama 3.3 70B at 30 requests/minute per API key. By using 4 keys with LiteLLM Router, IDOP effectively gets 120 requests/minute of free high-quality LLM inference — ideal for a production system without incurring per-token costs.

Error Handling & Common Issues

Error Handling Architecture

IDOP uses a structured error handling approach:

  • API endpoints: FastAPI HTTPException with status code + detail message
  • Graph nodes: Return error state (e.g., sql_status: "error", mutation_status: "error") — exceptions are caught and converted to state fields
  • Services: Standard Python exceptions, caught by calling code
  • Global: global_exception_handler catches unhandled exceptions at the FastAPI level

Common Errors & Solutions

Graceful Degradation

  • Vanna fails: Falls back to Groq LLM for SQL generation — user sees no difference
  • Redis unavailable: Falls back to in-memory cache — cache survives until restart
  • S3 unavailable: Falls back to local filesystem cache — developer mode
  • Database unavailable: Graph returns error state, approval tokens fall back to in-memory
  • LLM Judge fails: Defaults to is_correct=True — doesn't block valid queries
  • Embedding fails (quota): Retries 3 times with exponential backoff, then raises clear error
  • Router fails: Defaults to CHAT path — at minimum the LLM can still respond

Security Model

1. Cryptographic Approval Tokens

  • Every SQL generation and mutation upload generates a secrets.token_hex(32) token
  • Tokens are single-use — deleted from database after verification
  • Verified using secrets.compare_digest() — timing-safe comparison prevents timing attacks
  • Stored in PostgreSQL with in-memory fallback
  • Tokens are not returned in chat responses — only via explicit POST /sql/generate or GET /sql/pending

2. SQL Injection Prevention

  • All SQL goes through SQLValidator which blocks non-SELECT commands
  • Mutation SQL is fully parameterized — no string concatenation of user data
  • Word-boundary regex prevents false matches on column names (e.g., "updated_at" won't trigger UPDATE block)
  • LLM Judge provides a second opinion on semantic correctness

3. Mutation Safety

  • All mutations execute inside all-or-nothing transactions — if any row fails, all changes roll back
  • Rejected mutations (approved=false) are discarded from the pending queue with status "rejected"
  • Business rules validate each row before execution
  • DELETE operations always require explicit confirmation
  • Max row limit (1000) prevents resource exhaustion
  • Only supported tables can be mutated (configurable in schema_registry.py)

4. Audit Trail

  • Every SQL execution is logged to idop_audit_logs
  • Every mutation execution is logged to idop_audit_logs
  • Logs include: query_id, original question, SQL, status, timestamp
  • Both successful AND failed operations are logged

5. CORS

  • Wildcard origin (*) allowed by default
  • Credentials automatically disabled when wildcard is used (per CORS spec)
  • Specific origins can be configured via ALLOWED_ORIGINS

What Happens if Both Fail?

If both Vanna AND the Groq fallback fail (extremely rare — e.g., network outage), the SQL generation node returns:

{
  "sql_status": "error",
  "sql_explanation": "Generation failed: ...",
  "answer": "❌ SQL generation failed: ..."
}

This error message is returned to the user through the chat response. The LLM will say "Sorry, I couldn't generate SQL for that question."

How to Tell Which Path Was Used

Check the explanation field in the SQL generation response:

  • Vanna path: "SQL generated via Vanna"
  • Fallback path: "⚠️ Direct LLM Fallback: Generated using the primary LLM (LiteLLM/Groq) as the core Vanna agent was unavailable."
